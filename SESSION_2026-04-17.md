## Что точно работает

- **Пати 5/5 собирается стабильно**. Инвайты принимаются, `Party size=5` у всех 5 клиентов.
- **Ping_data captured** из `msg_type=8068`. В логе видно `[GC] Captured ping_data: 156 bytes`.
- **VEH + UnhandledFilter** установлены. В логе: `[CrashLog] VEH=0xNNN UnhandledFilter=prev=0xNNN`.
- **Server endpoint `/api/v1/dump` живой** (curl → 403 на неверный key, значит multipart парсится и auth проверяется).

## Что НЕ работает (→ в prompt_next)

1. **Попап "приглашение не актуально" не убирается после accept**. Пати собрана, member_ids содержит наш steamID, GC всё видит — но UI продолжает показывать попап.
2. **Авто-accept матча не срабатывает**. `ReadyUp` отправляется (с `hardware_specs`, правильным `ready_up_key`), но матч не стартует. **Если тестер вручную кликает Accept в ОДНОМ окне — в остальных 4 тоже принимается мгновенно**. Это сильная подсказка — см. предположения в prompt_next.
3. **Мидовый бот афк**. При удачном ручном запуске: 4 бота пошли на сайды, мид так и не двинулся.
4. **Скиллы после 2 lvl частично не качаются**. Lvl 1-2 берут, дальше пропуски.
5. **Сайдовые боты сходят с линии и стоят в лесу**.

## Ключевые находки (не потерять!)

### ESO msg_type canonical (gcsystemmsgs.pb.h)
```
21 = k_ESOMsg_Create
22 = k_ESOMsg_Update              ← БЫЛ пропущен!
23 = k_ESOMsg_Destroy
24 = k_ESOMsg_CacheSubscribed
25 = k_ESOMsg_CacheUnsubscribed
26 = k_ESOMsg_UpdateMultiple      ← БЫЛ пропущен!
```
Предыдущий enum путал 22↔23 и юзал 25 как "SOSingleObject". Это означало — любые обновления пати (add/remove members) уходили в `default:` switch.

### CMsgReadyUp схема (actual, не та что в dev_log 2026-03-17)
```proto
field 1 = state (DOTALobbyReadyState)
field 2 = ready_up_key (fixed64)     // lobby_id XOR ~(account_id | account_id<<32)
field 3 = hardware_specs (CDOTAClientHardwareSpecs) ← Обязательно!
```
`CDOTAClientHardwareSpecs`:
- f1 logical_processors
- f2 cpu_cycles_per_second
- f3 total_physical_memory
- f4 is_64_bit_os
- f5 upload_measurement
- f6 prefer_not_host
- f7 crc[]

### Поток ping_data
Клиент сам периодически шлёт `msg_type=8068 (k_EMsgClientToGCPingData)` — **body целиком = `CMsgClientPingData`**. Не ищем внутри его поля — забираем как есть и кладём в `mutable_ping_data()->ParseFromArray(...)` при accept/startFindingMatch.

### Poll thread = воровство сообщений
`RetrieveMessage` у Valve GC **consumes** сообщение из внутренней очереди. Если наш thread вызывает его сам — Dota UI больше этого сообщения не получит. Это был root cause "у 3 из 5 не видно пати". Теперь MinHook пассивно копирует каждый вызов Dota, ничего не добавляя.

### AcceptPartyInviteNative: handler is NULL
В `CGCMessages.cpp` RVA `NATIVE_ACCEPT_HANDLER_RVA=0x65599b0` → `*(void**)(clientBase + RVA) == NULL`. Указатель не инициализируется лениво или RVA устарел с билда 2026-03-25. **Это не починено** и напрямую связано с "попап не убирается" — native accept прошёл бы через UI path и cleanup'ил local state.

### Race: `joined_N.txt`
На втором раунде (после отмены матча) мемберы пишут joined пока они ещё в старой пати (GC SODestroy не пришёл). Лидер на следующем тике считает `joined==4` и **без отправки инвайтов** уходит в QUEUING. Фикс: мембер проверяет `gcState.partyID == ReadPartyId()` + лидер игнорирует файлы первые 3 сек FORMING_PARTY.

## Crash pipeline (новое в v0044-v0045)

```
Dota падает
  ↓
VEH / UnhandledExceptionFilter (в DLL)
  ↓
WriteCrashLogFile(regs, callstack, stack)  → debug_<PID>.log
WriteMiniDump(full_mem+threads)             → dump_<PID>_<YYYYMMDD>_<HHMMSS>.dmp
FlushFileBuffers()                          ← FILE_FLAG_WRITE_THROUGH тоже
  ↓
Chain to prev filter (Steam/WER)
  ↓
[60 сек позже: orchestrator.MonitorTick]
  telemetry::CollectLogs     → POST /api/v1/log   → /data/logs/<email>/
  скан *.dmp без *.sent
  telemetry::UploadDump      → POST /api/v1/dump  → /data/dumps/<email>/
  создаётся <dmp>.sent
```
