# Session 2026-04-18 — State refactor Phase A (GSI + расширенный status JSON)

## Цель сессии
Пользователь: "в меню боты не знают точно идёт ли поиск, в пати ли они, с кем в пати; in_game определяется только если инжект был в меню; нужен капитальный рефакторинг — сначала разведка, потом дизайн".

## Сделанная разведка

### Где данные сейчас (до сессии)
- DLL пишет плоский `C:\temp\andromeda\status_<PID>.json`: `state, hero, hp, maxHp, gameTime, gamesPlayed, instanceId, role`. Из всех богатых полей `gcState` (party/queue/match/lobby/cv/msgs) **наружу не передаётся ничего**.
- Orchestrator читает только эти 8 полей в `BotInstance`.
- Hot-inject GameRules восстанавливается через `CGameState::FullScan()` — работает. Но GC SOCache (party/lobby) теряется навсегда: `SOCacheSubscribed` приходит до инжекта.
- `OnEnter_InMenu` всегда слепо слал `LeaveParty()` — при hot-inject в живую пати разрушал её.

### Источники истины (что реально доступно)
1. **Dota 2 GSI** (нативный механизм Valve) — кладёшь cfg в `<dota>/game/dota/cfg/gamestate_integration/`, дота сама шлёт HTTP POST с полным in-game state. Работает только когда GameRules активен (т.е. в матче). Не нужен реверс, не нужна DLL.
2. **DLL `status_<PID>.json`** — единственный канал для menu/queue/party state (там где GSI молчит). Уже был, расширили схемой 2.
3. **CDOTAGCClientSystem singleton** в client.dll — потенциальный pull-источник для SOCache (party/lobby/invite) при hot-inject. Якоря найдены, но реализация — это Phase B.
4. **CNetworkGameClient** в engine2.dll — `m_eClientState` мог бы дать "соединён ли с сервером" без GC. Не используем (пока не нужно — GameRules+GSI закрывают).

### Якоря в client.dll (найденные строки и RTTI)
- `CDOTAGCClientSystem` string @ `client.dll+0x4e17df8` (1 hit, единственная)
- `.?AVCDOTAGCClientSystem@@` RTTI @ `client.dll+0x5f62510`
- `DOTAGCClientSystem` (interface name, без 'C') @ `client.dll+0x4e17df9`
- `.?AVCSharedObjectCache@GCSDK@@` RTTI @ `client.dll+0x6001c14` — namespace **GCSDK**

- type_id константы: `2003` = CSODOTAParty, `2004`/`2006` = CSODOTAPartyInvite, `2010` = CSODOTALobby
- `g_pCDOTAGCClientSystem` экспорта НЕТ → singleton хранится в FactorySystem (Source 2 паттерн)

## Что сделано (Phase A)

### A1 — GSI integration (orchestrator)
**Новые файлы:**
- `tools/dota2/orchestrator/src/gsi_server.h/.cpp` — Winsock HTTP listener на 127.0.0.1:3477, маппит snapshots по `player.steamid` → `provider.steamid`. Структура `GsiSnapshot`: matchId, gameState, hp/maxHp, mana, level, kda, gold, netWorth, position, alive, hero name.
- `tools/dota2/orchestrator/src/gsi_install.h/.cpp` — авто-находит Dota install (запущенный процесс → libraryfolders.vdf → стандартные пути) и кладёт `gamestate_integration_andromeda.cfg`.
- `tools/dota2/gsi_test_listener.py` — standalone Python listener для теста на этапе разработки.

**Изменения:**
- `CMakeLists.txt`: добавил `gsi_server.cpp`, `gsi_install.cpp`, линковка `ws2_32.lib`.
- `orchestrator.cpp`: вызов `EnsureGsi()` в `Init()` и в `StartFarmThread()` перед инжектом. `MonitorTick` зовёт `ReadGsiSnapshots()` ВСЕГДА (даже до RUNNING) — закрывает hot-inject.
- `orchestrator.h`: метод `EnsureGsi()`, `GetGsiStatus()`, поля `m_gsiCfgPath/m_gsiInstallError/m_gsiInstalled`.

**Cfg-файл (port 3477, throttle 0.1s, heartbeat 5s, token `andromeda_v1`)** — установлен на диске:

### A2 — расширенный status JSON (DLL → orchestrator)
**Изменения в DLL** (`Andromeda/BotFarm/CBotFarmClient.cpp`):
- `WriteStatusFile` теперь пишет **schema 2**:
  - `pid, instance_id, role, own_steam_id, heartbeat_ms`
  - `game_state` (полный `EDOTAGameState` строкой) + `game_state_id`
  - `sm: { state, state_seconds, auto_queue }`
  - `connection: { gc_ready, gc_msgs_total, client_version, ping_data_bytes }`
  - `party: { id, size, leader_steam_id, is_leader, member_steam_ids[], pending_invite_id }`
  - `queue: { in_queue, result_code }`
  - `match: { found, lobby_id }`
  - `game: { hero, hp, max_hp, alive, game_time, games_played }`
  - + старые плоские поля для backward-compat с schema 1
- Throttle уменьшен с 5s → **1s** для свежести данных.
- Используется `nlohmann/json` через `<Deps/json/json.hpp>`.

**Изменения в orchestrator**:
- `orchestrator.h::BotInstance` расширен: вложенные структуры `BotConn / BotParty / BotQueue / BotMatch`, поля `schema, gameState, gameStateId, smStateSeconds, alive, ownSteamId, heartbeatMs, heartbeatAgeMs, gsiSeen, gsi, gsiAgeMs`.
- `orchestrator.cpp::ReadStatusFiles` поддерживает оба schema (`jget` template helper). Считает `heartbeatAgeMs = now - heartbeat_ms`.

### A3 — фикс LeaveParty при hot-inject
`Andromeda/BotFarm/StateMachine/CBotStateMachine.cpp::OnEnter_InMenu`:
- Перестали слепо слать `LeaveParty()`.
- Теперь: если в текущей пати есть **хоть один не-наш** steam_id (не я и не из `arrPartyMembers`) — пати **сохраняется**. DEV_LOG пишет "keep party — hot-inject scenario".

### Дашборд (gui.cpp)
- **GSI status строка** сверху: `port | requests | seen steam_ids` (или красный warning).
- **Party panel**: группирует ботов по `party.id`, показывает `Party 0x... size=N leader=... | bots: #0* #1 #2`. Если в пати чужие — `+N non-bot`. Pending invites — отдельным блоком.
- **Bot table** расширена: добавлены `Lv`, `K/D/A`, `GSI` (age sec).
- **Operational status** (per-bot строка под таблицей): `#N DLL:Ns GC:msgs cv:N P:size LDR Q:Ns M:lobby <fullGameState>`. Цвета по возрасту heartbeat.

## Risks / unknowns
- GSI на **сборку из 5 ботов** на одной машине: все 5 шлют в один URL (один cfg), маппинг по `player.steamid`. Концептуально работает, но в live-multibox не тестировалось. Стоит пройти прогон через `Start Farm`.
- Schema 2 в DLL пишется через `nlohmann::json` — раньше использовался `snprintf`. Если есть аномалии с включением `<Deps/json/json.hpp>` в `CBotFarmClient.cpp` (linker bloat / ICE) — откатить на ручной snprintf для верха словаря.
- `EnsureGsi` зовётся из `Init()` ДО `gui::Init()`. Если порт 3477 занят (наш Python listener из теста) — orchestrator упадёт молча через `m_gsiInstallError`. В live прогоне Python надо остановить (TaskStop) — уже сделано в этой сессии.
